# Quiz Generator

ユーザーが指定したソースファイルから、5択クイズのHTMLツールを自動生成してGitHub Pagesにデプロイするスキル。

## 使い方

ユーザーに以下を確認する：
1. **ソースファイル** — PDF、Markdown、テキストファイルのパス（複数可）
2. **クイズのタイトル** — 例：「Sアセス一次試験 対策クイズ」
3. **出力ファイル名** — 例：`s-assess-quiz.html`
4. **カテゴリ分け** — ソースごとにカテゴリを分けるか、自動判定か

## 実行フロー

### Step 1: ソーステキスト抽出

各ファイルからテキストを抽出する：
- **PDF**: `pdftotext "ファイルパス" -` で抽出（poppler未インストールなら `brew install poppler`）
- **Markdown**: そのまま読む
- **テキスト**: そのまま読む
- 大きなPDF（10MB超）はスキャン画像の可能性あり。テキスト抽出結果が空や極端に短い場合はユーザーに報告

### Step 2: 問題生成

抽出したテキストから5択問題を生成する。**絶対ルール：**

- **ソースに書かれている事実のみ**から問題を作る。推測・補完・一般知識で問題を作ってはならない
- 各問題は**5つの選択肢、正解は必ず1つ**
- 正解以外の4つの選択肢（ディストラクター）は**もっともらしいが明確に誤り**であること
- 複数正解になりうる曖昧な問題は作らない
- 各問題に**解説**（なぜその答えが正しいか）と**出典**（どのソースの何の項目か）を付ける

**問題の種類：**
- 定義問題：「〜とは何か」「〜の説明として正しいものは」
- 数値問題：「〜の値として正しいものは」
- 計算問題：「〜の場合、〜はいくらか」（計算式と答えを解説に含める）
- 正誤判定：「〜として誤っているものはどれか」
- 分類問題：「〜に含まれないものはどれか」

**品質基準：**
- 正答インデックス（0-4）の分布を均等にする（各20%±5%）
- 同じ正答インデックスが4回以上連続しない
- 「上記のうち…」系のメタ選択肢は使わない
- 「上記の条件で…」のような前の問題に依存する表現は使わない（ランダム出題時に意味不明になる）
- 最長の選択肢が正解になるパターンを避ける

**目標問題数：** ソースの分量に応じて50-200問。少なすぎるより多い方がよい。

### Step 3: HTML生成

以下の機能を持つスタンドアロンHTMLファイルを生成する：

```
構成:
- ヘッダー（タイトル、説明）
- スタッツバー（出題数、正解数、不正解数、正答率）
- モード切替（ランダム / 苦手重点 / 順番出題）
- カテゴリフィルター
- 問題カード（問題文、5択ボタン、結果表示、次へボタン）
- 学習記録パネル（累計、カテゴリ別正答率、セッション履歴）
- 回答履歴パネル（直近10問）
- 結果エクスポート/リセットボタン
```

**必須機能：**
- **localStorage永続化** — 回答するたびに自動保存。リロードしても記録が残る
  - 累計スタッツ（出題数、正解数、不正解数）
  - 問題別正誤履歴（各問題のインデックス → 試行回数、正解数、不正解数、最終回答日時）
  - カテゴリ別スタッツ
  - セッション履歴（開始時刻、回答数、正解数）直近20件
  - 苦手問題インデックスリスト
- **苦手重点モード** — 間違えた問題を70%の確率で再出題。正解したら苦手リストから除外
- **ランダムモード** — 全問出題し終わるまで重複なし。一周後にリセット
- **順番出題モード** — フィルタされた問題を順番に出題
- **カテゴリフィルター** — カテゴリ別に出題範囲を絞れる
- **結果エクスポート** — カテゴリ別成績と苦手問題リストをクリップボードにコピー
- **ページ離脱時のセッション保存** — beforeunloadでセッションデータを保存

**デザイン：**
- モバイルファースト（iPhone Safariでの利用を想定）
- フォント: `-apple-system, BlinkMacSystemFont, 'Hiragino Sans'`
- タッチターゲット: 最低44px
- viewport meta tag必須
- CSS変数でカラーテーマ管理

### Step 4: 品質監査

生成後に必ず以下を検証する。**サブエージェントを並列で3つ起動**して監査：

1. **正誤チェック** — 全問題の answer インデックスが正しい選択肢を指しているか。解説が正答と矛盾していないか
2. **ソース照合** — 問題の事実がソース資料と一致しているか。数値・定義・名称に誤りがないか
3. **出題品質** — 正答分布の均等性、曖昧な選択肢、明らかに不自然なディストラクター、重複問題がないか

監査で問題が見つかったら修正する。

### Step 5: デプロイ

1. 出力HTMLをワーキングディレクトリに保存
2. `git add` → `git commit` → `git push`
3. GitHub Pagesが有効か確認。未設定なら `gh api` で設定
4. デプロイ完了を確認してURLを返す

## HTMLテンプレート（参考）

問題データの形式：
```javascript
const questions = [
  {
    category: "カテゴリ名",
    question: "問題文",
    choices: ["選択肢A", "選択肢B", "選択肢C", "選択肢D", "選択肢E"],
    answer: 0,  // 0-indexed
    explanation: "解説テキスト",
    source: "出典情報"
  },
  // ...
];
```

## 注意事項

- ハルシネーション厳禁。ソースに書いていないことは問題にしない
- 選択式問題で「他にも正解がありえる」状態は絶対に避ける
- 計算問題の答えは必ず自分で検算する
- ユーザーのフィードバックがあれば優先的に反映する
